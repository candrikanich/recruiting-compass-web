Permission Functions (server/utils/auth.ts)
Four permission utilities — called inside API handlers
1
requireAuth(event)
Bearer header → cookie fallback → supabase.auth.getUser(token). Returns AuthUser or throws 401.
2
getUserRole(userId, supabase)
Reads users.role. In-memory cache: 5-min TTL, max 1,000 entries (LRU eviction). Returns "player" | "parent" | null.
3
assertNotParent(userId, supabase)
Throws 403 "Parents cannot perform this action" if role = "parent". Called on all mutation endpoints.
4
canMutateAthleteData(reqUserId, targetUserId, sb)
true only if requestingUser === targetAthlete. Parents always false. Cross-user mutations always false.
5
requireAdmin(event)
requireAuth() + DB check users.is_admin = true. 403 if not admin. Logs attempted violations.
6
isParentViewingLinkedAthlete(parentId, athleteId, sb)
Checks account_links WHERE status = "verified". Used for read-permission gates.

iOS Native API Auth — CSRF Exemption
⚠ Why iOS is exempt from CSRF
CSRF attacks exploit cookies. The iOS app sends only a Bearer token — no sb-access-token cookie.

Detection in csrf.ts:
isBearerOnly = hasAuthHeader AND !hasCookie if (isBearerOnly) return // skip CSRF check
Web browsers set both Bearer header (useAuthFetch) AND cookie (Supabase SDK) → full CSRF enforcement applies.
